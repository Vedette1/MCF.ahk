# MCF (MCode Forge)

MCF (MCode Forge) is a standalone COFF linker implemented in AutoHotkey v2, designed specifically for the AutoHotkey v2 platform.

![Screenshot](https://github.com/Vedette1/MCF.ahk/blob/main/screenshot1.png)

## About the Project

The main goal of the project is to simplify building MCode from C source code. Write plain C code and get the finished MCode with a single click.

## Architecture and Linker Operation Principle (COFF Class)

The core of the project is a custom linker (the `COFF` class) implemented in pure AutoHotkey v2. It parses and assembles COFF (Common Object File Format) object files generated by compilers (GCC/Clang, MSVC) and converts them into ready-to-use machine code (`MCode`) without requiring third-party software.

1.  **Parsing Header Structure:** The linker reads the file headers (`IMAGE_FILE_HEADER`), section table (`IMAGE_SECTION_HEADER`), and symbol table (`IMAGE_SYMBOL`). At this stage, unnecessary auxiliary sections are filtered out (the linker skips sections with the following characteristics by default: `IMAGE_SCN_LNK_REMOVE`, `IMAGE_SCN_LNK_INFO`, `IMAGE_SCN_MEM_DISCARDABLE`). Additionally, you can specify sections (by name) to be ignored during linking (by default these are: `.pdata`, `.xdata`, `.rdata$zzz`).
2.  **Layout Formation (Memory Map):** Sections are arranged sequentially into a single continuous buffer, respecting memory alignment from the characteristics (`IMAGE_SECTION_HEADER.Characteristics`), and new offsets are calculated for each section.
3.  **Resolving Relocations (Address Patching):**
    * **Internal Relocations:** The relocation table (`IMAGE_RELOCATION`) is traversed. Instruction offsets within the code are automatically adjusted (see the `ApplyRelocation` method for all supported relocation types) to ensure correct cross-section interaction of variables and functions in the resulting buffer.
    * **External Calls (Imports):** The linker finds external symbols (e.g., WinAPI function calls via `__imp_`). Instead of generating a full classical PE import table (`IAT`), the project uses an optimized solution — a custom compact **IAT tail** is formed (it looks approximately like this: `|msvcrt:isspace:51:4` - `[|dllName:funcName:reloc:relocType|...]`), containing metadata about the DLL, function names, and patching offsets.
4.  **Symbol Table Generation:** The linker exports an offset map (for functions, global variables, and sections), allowing you to find the entry point offset of specific C/C++ functions within the resulting machine code block.

## Graphical User Interface (GUI)

The project is equipped with a convenient graphical interface designed as a **micro-IDE** for ease of coding if you don't have a proper IDE.

### Key GUI Features:
* **Integrated Mini-IDE:** The text area supports syntax highlighting for C/C++ source code, as well as for `objdump` disassembled output (currently only Intel syntax is supported).
* **Complete Autonomy (Zero Dependencies):** The operation of the GUI and the built-in linker is **not rigidly tied** to a compiler or the `objdump` utility. They are merely optional auxiliary tools for automating compilation and providing debugging information in the form of a disassembler.
* **Building from Ready-Made Files:** If you don't have a compiler installed on your system, you can simply specify the path to any existing object file (`.o` / `.obj`), and the built-in linker will successfully assemble the `MCode` without any external dependencies.
* **Flexible Export:** The linking result can be obtained in `HEX`, `Base64`, or compressed (`Compressed Base64` via built-in `ntdll.dll` compression algorithms) formats with a single click.

## Example
```ahk
/**
#include <windows.h>
int msgBox() {
	return MessageBox(0, "Hello Mcode", "Title", 0);
}
*/
; compiler : TDM-GCC 10.3.0
; flags    : -m64 -O2 -fPIC -fno-stack-protector -fno-plt -fno-asynchronous-unwind-tables

offset := Map()
offset["msgBox"] := 0x0 ; 2
offset[".text"] := 0x0 ; 4 [dispensable]
offset[".data"] := 0x20 ; 6 [dispensable]
offset[".bss"] := 0x20 ; 8 [dispensable]
offset[".rdata"] := 0x20 ; 10 [dispensable]

ptr := GetMcodePtr("4531c94c8d0516000000488d151500000031c9ff2500000000909090909090905469746c650048656c6c6f204d636f6465000000000000000000000000000000|User32:MessageBoxA:21:4") ; Copy the GetMcodePtr function from the GUI.
DllCall(ptr, "int")
```

## Current Limitations

* **CRT (C Runtime):** The linker does not support the C Runtime Library (CRT). Code using standard C functions like `printf` cannot currently be processed correctly. The reason is that the CRT requires linking with static libraries, for which a parser has not yet been implemented. For example, if you see **strange** function names that are clearly not in the source code (`__acrt_iob_func`, `__mingw_vfprintf`, and the like), your `Mcode` will not work.
* **SEH (Structured Exception Handling):** Exception handling (`try / catch`) in x64 requires registering exception tables (`.pdata` and `.xdata`) with the Windows kernel via `RtlAddFunctionTable`. I have absolutely no idea how to do this, although I haven't even tried to figure it out.
